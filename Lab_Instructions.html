Lab 1: Implement Secure Data Communication with AES and RSA

Generate modern RSA keys, wrap an AES key with OAEP, encrypt data with AES-256-CBC, and verify integrity/cleanup.

Step 1: Generate RSA Keys (2048-bit)

Use the supported pkey/pkeyutl commands instead of deprecated rsautl.

# Generate a 2048-bit RSA private key openssl genpkey -algorithm RSA -out private.pem -pkeyopt rsa_keygen_bits:2048 # Derive the public key from the private key openssl pkey -in private.pem -pubout -out public.pem # Restrict permissions on private key (POSIX) chmod 600 private.pem

Step 2: Wrap a 256-bit AES Key with RSA-OAEP

Generate a binary AES key file, then encrypt it with the public key using OAEP padding.

# Generate a random 256-bit AES key (binary) openssl rand -out aes.key 32 # Encrypt (wrap) the AES key using RSA-OAEP openssl pkeyutl -encrypt -inkey public.pem -pubin -in aes.key \ -pkeyopt rsa_padding_mode:oaep -out aes.key.enc

Step 3: Encrypt a File with AES-256-CBC

Use the generated key material, derive IV from random bytes, and store non-secret metadata alongside ciphertext.

# Create a sample plaintext file echo "Hello, World!" > plaintext.txt # Generate a random IV (16 bytes) openssl rand -out iv.bin 16 # Encrypt using the binary key and IV openssl enc -aes-256-cbc -K $(xxd -p aes.key) -iv $(xxd -p iv.bin) \ -in plaintext.txt -out ciphertext.enc

Step 4: Unwrap and Decrypt

Recover the AES key with the private key, decrypt the file, and verify integrity.

# Decrypt (unwrap) the AES key with the private key openssl pkeyutl -decrypt -inkey private.pem -in aes.key.enc \ -pkeyopt rsa_padding_mode:oaep -out aes.key.dec # Decrypt using the recovered key and stored IV openssl enc -d -aes-256-cbc -K $(xxd -p aes.key.dec) -iv $(xxd -p iv.bin) \ -in ciphertext.enc -out decrypted_ciphertext.txt # Verify content matches diff plaintext.txt decrypted_ciphertext.txt # (Optional) Generate and verify a SHA-256 checksum sha256sum plaintext.txt decrypted_ciphertext.txt

Step 5: Cleanup & Safety

  • Shred keys after use in shared labs: shred -u private.pem aes.key*.
  • Keep iv.bin and aes.key.enc with ciphertext; keep private.pem offline.
  • Document who holds each key and rotate keys for new exercises.

Lab 2: Configure Access Control Policies in a Database

Create sample tables, roles, and grants in MySQL or PostgreSQL; verify and revoke privileges; inspect audit events.

Step 1: Create a Database & Tables

-- MySQL example CREATE DATABASE big_data_project; USE big_data_project; CREATE TABLE patient_records ( id INT PRIMARY KEY AUTO_INCREMENT, name VARCHAR(50), diagnosis VARCHAR(100) ); CREATE TABLE transaction_logs ( id INT PRIMARY KEY AUTO_INCREMENT, actor VARCHAR(50), action VARCHAR(50) ); INSERT INTO patient_records (name, diagnosis) VALUES ('Ada', 'Hypertension'), ('Grace', 'Migraine');

Step 2: Define Roles and Users

Use roles to group privileges; then assign roles to users.

-- Create roles CREATE ROLE analyst; CREATE ROLE data_engineer; CREATE ROLE admin; -- Create users CREATE USER 'analyst_user'@'%' IDENTIFIED BY 'P@ssw0rd!'; CREATE USER 'data_engineer_user'@'%' IDENTIFIED BY 'P@ssw0rd!'; -- Assign roles to users GRANT analyst TO 'analyst_user'@'%'; GRANT data_engineer TO 'data_engineer_user'@'%';

Step 3: Grant & Verify Privileges

Grant privileges to roles (not users) and test with SET ROLE.

-- Grants to roles GRANT SELECT ON patient_records TO analyst; GRANT INSERT, UPDATE, DELETE ON transaction_logs TO data_engineer; GRANT ALL PRIVILEGES ON *.* TO admin WITH GRANT OPTION; -- Validate as analyst SET ROLE analyst; SELECT * FROM patient_records; -- succeeds INSERT INTO patient_records VALUES (999, 'Test', 'Should fail'); -- should fail -- Validate as data_engineer SET ROLE data_engineer; INSERT INTO transaction_logs (actor, action) VALUES ('engineer', 'import'); SELECT * FROM transaction_logs; -- should succeed

Step 4: Revoke & Audit

-- Revoke UPDATE for data_engineer role REVOKE UPDATE ON transaction_logs FROM data_engineer; -- MySQL audit (table-based general log) SET GLOBAL log_output = 'TABLE'; SET GLOBAL general_log = 'ON'; SELECT event_time, user_host, argument FROM mysql.general_log WHERE command_type = 'Query' ORDER BY event_time DESC LIMIT 5; -- PostgreSQL note: enable pgaudit extension and query pgaudit.log
  • Document least-privilege decisions (why each grant exists).
  • Show how to disable logging after the lab: SET GLOBAL general_log = 'OFF';
  • Rotate passwords or drop test users when finished.

Lab 3: Implement Multi-Factor Authentication (MFA)

Create a virtual environment, onboard a TOTP secret with a QR code, and verify codes with drift/rate-limit protections.

Step 1: Isolate & Install Dependencies

python3 -m venv .venv source .venv/bin/activate pip install --upgrade pip pip install pyotp qrcode[pil]

Step 2: Provision a User Secret

Generate a random Base32 secret per user and produce a provisioning URI for authenticator apps.

import pyotp, qrcode # Generate per-user secret secret = pyotp.random_base32() totp = pyotp.TOTP(secret) # Provisioning URI (scan with Google Authenticator / Authy) uri = totp.provisioning_uri(name="student@example.com", issuer_name="AD23631-MFA-Lab") qrcode.make(uri).save("totp-qr.png") print("Secret:", secret) print("URI saved to totp-qr.png")

Step 3: Verify with Clock Drift Allowance

Accept codes within one window for minor skew and implement basic retry limits.

import time MAX_ATTEMPTS = 3 attempts = 0 while attempts < MAX_ATTEMPTS: code = input("Enter the 6-digit code: ") if totp.verify(code, valid_window=1): # allows +/-30s drift print("Authentication Successful") break attempts += 1 print("Invalid code. Attempts left:", MAX_ATTEMPTS - attempts) else: print("Too many failures ‚Äî account locked, alert security team")

Step 4: Secure Secret Handling

  • Store secrets in environment variables or a secrets manager, not source control.
  • Rotate secrets on device loss and enforce per-session nonce/state to prevent replay.
  • Deactivate the virtual environment after testing: deactivate.

Lab 4: Case Study - Capital One Data Breach

Build a timeline, map controls, and run a tabletop drill based on the 2019 Capital One breach.

Incident Overview

  • Target: Capital One (2019)
  • Impact: 100M+ customers exposed.
  • Root causes: SSRF exploit + misconfigured WAF & IAM roles.

Step 1: Build a Timeline

  • Collect events: vulnerability exploit, data exfil, detection, response, disclosure.
  • Record sources (CloudTrail, WAF logs) and note detection gaps.

Step 2: Map Controls to Frameworks

  • Map missing controls to NIST CSF (Identify/Protect/Detect/Respond/Recover).
  • Propose fixes: strict IAM roles, instance metadata protection, patched WAF rules, VPC egress limits.

Step 3: Tabletop Drill Script

  • Roles: Incident Commander, Comms Lead, Cloud Engineer, Security Analyst.
  • Decision points: contain access keys, rotate credentials, isolate workloads, notify regulators.
  • Deliverables: evidence log, owner/action/deadline table, post-incident hardening checklist.

Step 4: Postmortem & Hardening

  • Document blast radius, data classification, and customer impact.
  • Implement compensating controls: IMDSv2, rotate keys, least-privilege WAF/IAM, alerting on anomalous role usage.
  • Schedule retests and track owners in a playbook template.

Lab 5: Cloud IAM (AWS)

Set up a sandbox account, create IAM users/groups/policies via CLI, validate permissions, and clean up safely.

Step 1: Sandbox & CLI Setup

  • Use a training/sandbox AWS account to avoid production risk.
  • Install/verify AWS CLI: aws --version, then aws configure with sandbox keys.
  • Enable MFA on the root account before proceeding.

Step 2: Define a Least-Privilege Policy

{ "Version": "2012-10-17", "Statement": [ { "Sid": "AllowS3ReadAccess", "Effect": "Allow", "Action": ["s3:GetObject", "s3:ListBucket"], "Resource": [ "arn:aws:s3:::my-secure-bucket", "arn:aws:s3:::my-secure-bucket/*" ] } ] }

Step 3: Create Groups, Users, and Attach Policy

# Create a group and attach policy aws iam create-group --group-name DataAnalysts aws iam create-policy --policy-name S3ReadOnly --policy-document file://s3-read.json aws iam attach-group-policy --group-name DataAnalysts \ --policy-arn arn:aws:iam:::policy/S3ReadOnly # Create users and add to group aws iam create-user --user-name analyst-user aws iam add-user-to-group --user-name analyst-user --group-name DataAnalysts # Create access keys for the user (store securely!) aws iam create-access-key --user-name analyst-user

Step 4: Validate & Simulate

# Verify identity and permissions aws sts get-caller-identity aws iam get-user --user-name analyst-user # Test S3 access with the user's credentials AWS_PROFILE=analyst aws s3 ls s3://my-secure-bucket # IAM Policy Simulator (console) or CLI example aws iam simulate-principal-policy --policy-source-arn \ arn:aws:iam:::user/analyst-user --action-names s3:GetObject

Step 5: Cleanup & Rotation

  • Rotate access keys every 90 days; prefer federation with MFA for long-term use.
  • Teardown after the lab: detach policies, remove keys, delete users/groups.
  • Confirm CloudTrail is on; review access logs for the lab period.

Lab 6: Differential Privacy

Implement correct Laplace noise, bound sensitivity, and observe privacy/utility trade-offs.

Step 1: Define the Query & Sensitivity

Example: class average. Bound each student's score to [0, 100] so L1 sensitivity of the sum is 100; for the average, sensitivity = 100 / n.

Step 2: Implement Laplace Mechanism

Scale = sensitivity / epsilon (Œ¥ not used for Laplace Œµ-DP).

import numpy as np def laplace_mech(true_answer, sensitivity, epsilon): scale = sensitivity / epsilon noise = np.random.laplace(loc=0, scale=scale) return true_answer + noise # Example usage scores = [88, 92, 77, 95, 84] bounded = [min(100, max(0, s)) for s in scores] true_avg = sum(bounded) / len(bounded) sensitivity = 100 / len(bounded) epsilon = 1.0 noisy_avg = laplace_mech(true_avg, sensitivity, epsilon) print(f"True: {true_avg:.2f}, Noisy: {noisy_avg:.2f}")

Step 3: Explore Composition & Accuracy

  • Run the query repeatedly; track total Œµ budget spent.
  • Plot error distributions for Œµ = 0.1, 0.5, 1, 5 to visualize utility.
  • Discuss clipping/normalization choices and their impact on sensitivity.
